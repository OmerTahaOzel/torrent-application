

==================================================
1) PROJE TANIMI (SLIDE 1)
==================================================

Proje adi:
- Torrent Application

Teknik kapsam:
- Dil/Platform: C# / .NET 8
- UI: WinForms
- Ag: TCP socket
- Merkezi bilesen: Tracker

==================================================
2) NEDEN BU MIMARI? (SLIDE 2)
==================================================

Mimari parcalar:
1. torrent application (istemci)
2. Torrent.Core (is mantigi)
3. Torrent.Tracker (peer discovery servisi)

Neden bu ayrim secildi?
- UI degisse bile core mantik bozulmasin.
- Tracker'i ayri process calistirarak dagitik sistemi basit sekilde simule edelim.
- Test ve bakimi kolaylastiralim.

==================================================
3) KULLANICI EKRANI UZERINDEN AKIS (SLIDE 3)
==================================================

Node + Tracker paneli:
- Tracker host/port
- Advertised host
- Listen port
- Node Baslat / Auto Port

Seeder sekmesi:
- Dosya sec
- Seeding baslat
- .ttmeta olustur

Leecher sekmesi:
- .ttmeta sec
- Cikis klasoru sec
- Indir
- Progress + log takip

Kapanis davranisi:
- Indirme devam ederken pencere kapanirsa once iptal edilir, sonra kapanir.

==================================================
4) TORRENT MANTIGI - DOSYA NASIL HAZIRLANIYOR? (SLIDE 4)
==================================================

Seeding baslatinca olanlar:
1. Dosya 1 MB parcalara bolunur.
2. Her parca SHA-256 ile hashlenir.
3. Parca hashlerinden Merkle tree olusturulur.
4. Merkle root + parca hashleri metadata'ya yazilir.
5. .ttmeta dosyasi olusur.

Neden parca hash + Merkle birlikte?
- Parca hash: tek parcayi dogrular.
- Merkle root/proof: parcayi tum dosya butunlugune baglar.

==================================================
5) TRACKER NASIL CALISIYOR? (SLIDE 5)
==================================================

Tracker gorevi:
- Dosya tasimaz.
- Sadece "bu infoHash hangi peer'lerde var" bilgisini tutar.

Sunumda vurgulanacak nokta:
"Tracker veri aktarmiyor; sadece adres dagitiyor. Veri P2P akiyor."

==================================================
6) DOWNLOAD AKISI (SLIDE 6)
==================================================

Leecher'da Indir deyince:
1. .ttmeta okunur.
2. Tracker'dan peer listesi alinir.
3. Peer'lere handshake atilir.
4. Bitfield bilgisi toplanir.
5. Peer'lerden parcalar cekilir.
6. Her parca verify edilir.
7. Dogru parca dosyaya yazilir ve state kaydedilir.
8. Tum parcalar bitince final Merkle root verify edilir.

Hata olursa:
- Parca basarisizsa yeniden denenecek sekilde queue'ya geri birakilir.

==================================================
7) GUVENILIRLIK: RESUME VE CANCELLATION (SLIDE 7)
==================================================

Resume:
- .ttstate dosyasi tamamlanan parcalari tutar.
- Uygulama kapanip acilinca indirme sifirdan degil kaldigi yerden devam eder.

Cancellation:
- Form kapanirken aktif indirme varsa kontrollu iptal edilir.
- Task bitmeden form tamamen kapanmaz.

Neden onemli?
- Demo ve gercek kullanimda veri kaybi/bozukluk riskini dusurur.

==================================================
8) PROTOKOL TASARIMI (SLIDE 8)
==================================================

Wire format:
- 4 byte length prefix + JSON payload.

Avantajlari:
- Basit, okunabilir, debug etmesi kolay.
- Socket stream'de paket siniri problemini cozer.

Mesaj tipleri:
- Handshake
- Bitfield
- RequestPiece
- PieceData
- MetadataRequest / MetadataResponse
- Error

==================================================
9) SONUC (SLIDE 9)

==================================================
EK: SUNUMDA GELEBILECEK SORULARA HAZIR CEVAPLAR
==================================================

Soru: Neden tracker merkezi, bu p2p degil mi?
Cevap: Veri akisi p2p, sadece peer discovery merkezi. Bu egitsel projede karmasikligi azaltmak icin secildi.

Soru: Neden Merkle kullandin, tek hash yetmez miydi?
Cevap: Tek final hash sadece en sonda kontrol verir. Merkle proof ile parca gelir gelmez dogrulama yapabiliyoruz.

Soru: Neden JSON kullandin?
Cevap: Okunabilirlik ve debug kolayligi. Egitim projesinde binary protokole gore daha anlatilabilir.

Soru: Heartbeat olmasa olur mu?
Cevap: Olur ama stale peer kalma riski artar. Bu projede daha gercekci davranis icin eklendi.
